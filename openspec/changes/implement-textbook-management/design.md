# Design: Textbook Management

## Context

### 知识库存储模式

数学通采用**基于文件的知识库管理**模式：

1. **文件存储位置**: `iksm/` 目录（IKSM - Intelligent Knowledge System for Math）
2. **文件配对规则**: 每个教材由一对文件组成
   - 框架文件: `{name}.xlsx` 或 `{name}.csv` - 知识点层级结构
   - 详情文件: `{name}.md` - 知识点的详细 Markdown 内容
3. **示例**: `math01.xlsx` + `math01.md` 构成完整教材

### 知识点数据结构

框架文件（CSV/Excel）包含以下字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| 一级知识点 | 大章节名称 | 集合与常用逻辑用语 |
| 二级知识点 | 小节名称 | 集合的概念与表示 |
| 三级知识点 | 具体知识点 | 集合的含义 |
| 定义 | 知识点定义描述 | 研究对象的总体 |
| 特性/运算方式 | 特性或运算方法 | 确定性、互异性、无序性 |
| 知识点编号 | 唯一标识符 | 1.1.1 |
| 重要性级别 | A/B/C 分类 | A类=必须掌握 |

### 当前限制

- 文件格式：仅支持 `.xlsx`, `.csv`, `.md`
- 文件命名：以 `.`、`_`、`tmp` 开头的文件会被忽略
- 文件关联：通过文件名前缀匹配（如 `math01`）

## Goals

### 必须实现 (Must Have)

1. **文件管理**
   - [ ] 扫描 `iksm/` 目录，识别所有教材文件对
   - [ ] 提供文件树视图，展示目录结构
   - [ ] 支持文件上传（框架文件 + 详情文件）
   - [ ] 支持文件删除

2. **知识点解析**
   - [ ] 解析 Excel (.xlsx) 文件
   - [ ] 解析 CSV (.csv) 文件
   - [ ] 提取知识点层级结构（level1/level2/level3）
   - [ ] 验证知识点数据完整性

3. **数据同步**
   - [ ] 检测文件变更（新增、修改、删除）
   - [ ] 将知识点数据同步到 PostgreSQL
   - [ ] 记录文件最后修改时间，用于变更检测

4. **用户界面**
   - [ ] 左侧文件树 + 右侧列表的双栏布局
   - [ ] 教材列表展示（名称、文件类型、状态、操作）
   - [ ] 上传对话框（支持拖拽）
   - [ ] 同步状态提示

### 期望实现 (Nice to Have)

1. **批量上传** - 同时上传多个教材文件对
2. **导入预览** - 上传前预览解析结果
3. **冲突处理** - 文件名冲突时的处理策略
4. **历史版本** - 保留教材文件的历史版本

### 非目标 (Non-Goals)

1. 不实现 Markdown 内容的实时编辑（在知识点学习页面处理）
2. 不实现文件版本控制和回滚
3. 不支持除 xlsx/csv/md 外的其他文件格式

## Decisions

### 1. 文件解析库选择

**决策**: 使用 `xlsx` + `papaparse`

**选项对比**:

| 库 | Excel | CSV | 优点 | 缺点 |
|----|-------|-----|------|------|
| xlsx + papaparse | ✅ | ✅ | 成熟稳定，社区活跃 | 需要两个库 |
| exceljs | ✅ | ❌ | 功能丰富 | 体积大，CSV需额外处理 |
| fast-csv | ❌ | ✅ | 性能好 | 不支持 Excel |

**理由**:
- `xlsx` 是解析 Excel 的标准库，支持浏览器和 Node.js
- `papaparse` 是 CSV 解析的首选，支持流式解析
- 两者组合满足所有需求，且社区维护活跃

### 2. 文件同步策略

**决策**: 基于文件修改时间的增量同步

**策略细节**:
```
1. 扫描 iksm/ 目录，获取所有文件及修改时间
2. 对比数据库记录的文件修改时间
3. 只同步有变更的文件，避免全量重建
4. 同步时采用事务：删除旧知识点 → 插入新知识点
```

**替代方案**: 基于文件内容哈希
- 优点：能检测内容相同但修改时间变化的文件
- 缺点：计算哈希开销大，收益有限

**理由**: 修改时间检测简单高效，符合教材文件更新频率低的场景

### 3. 文件上传方式

**决策**: 前端直传 + 后端验证

**流程**:
```
1. 前端选择/拖拽文件
2. 前端验证文件类型（xlsx/csv/md）
3. 上传到后端临时目录
4. 后端验证文件格式和内容
5. 移动到 iksm/ 目录
6. 触发同步流程
```

**理由**:
- 即时反馈上传进度
- 后端验证确保数据质量
- 临时目录机制防止写入不完整文件

### 4. 数据库同步粒度

**决策**: 教材级别全量替换

**实现**:
- 同步单个教材时，删除该教材关联的所有知识点
- 重新解析并插入新的知识点数据
- 使用数据库事务保证一致性

**理由**:
- 教材文件是一个整体，部分更新逻辑复杂
- 知识点数量可控（单个教材通常 < 500 条）
- 简化实现，避免增量更新的复杂逻辑

### 5. 文件树展示策略

**决策**: 扁平化展示教材列表 + 文件类型分组

**替代方案**: 真实目录树
- 优点：符合文件系统结构
- 缺点：当前需求只需要识别教材文件对，无需深层目录

**实现**:
```
iksm/
├── math01.xlsx + math01.md  →  教材：高中数学必修一
├── math02.xlsx + math02.md  →  教材：高中数学必修二
└── physics01.csv + physics01.md → 教材：高中物理必修一
```

**理由**: 简单直观，符合当前使用场景

## Risks / Trade-offs

### 1. 大文件处理

**风险**: Excel/CSV 文件可能很大，解析耗时

**缓解策略**:
- 使用流式解析（papaparse 支持）
- 异步处理，提供进度反馈
- 设置文件大小限制（如 10MB）

### 2. 并发同步

**风险**: 多个教师同时操作可能导致冲突

**缓解策略**:
- 教材文件使用唯一文件名
- 同步操作使用数据库事务
- 添加操作锁或队列（如果需要）

### 3. 数据一致性

**风险**: 文件系统与数据库可能不一致

**缓解策略**:
- 提供手动刷新功能
- 定期自动扫描（可选）
- 清晰的同步状态指示

### 4. 文件格式兼容性

**风险**: 不同版本的 Excel 文件解析可能失败

**缓解策略**:
- 支持 .xlsx 和 .csv 两种格式作为备选
- 提供清晰的错误信息和修复建议
- 文档中明确支持的格式版本
